# Cursor Client Implementation

## Quick Start

当任务涉及功能实现、交互调整、工作流优化或类似 Cursor 的客户端界面时：

1. 先在当前仓库定位相关模块、状态流和 UI 入口。
2. 再到 `D:/WebstormProjects/claude-code/claude-code-2.1.88/src` 查找类似实现。
3. 提炼可迁移的结构、交互和边界处理，不直接照抄。
4. 落地时同时满足当前仓库约定和 Cursor 风格交互。

## Required Workflow

### 1. 先做对照搜索

- 先找当前仓库的相关代码，再找外部参考仓库的类似功能。
- 优先对照这些点：
  - 组件拆分方式
  - 状态来源与事件流
  - 异步或流式更新
  - 加载态、空态、错误态
  - 键盘操作与焦点管理
  - 面板、弹窗、输入框、列表和命令式入口
- 如果外部仓库存在近似实现，在开始编码前先总结值得复用的模式。

### 2. 适配，不要硬拷贝

- 复用模式，不复用与当前仓库不兼容的命名、依赖或层级。
- 保持当前项目已有的目录结构、命名风格、状态管理方式和类型边界。
- 如果外部实现与当前架构冲突，优先遵守当前仓库约定，并保留参考思路。

### 3. 默认采用 Cursor-like 交互

优先使用这些交互原则：

- 键盘优先，尽量支持快捷操作、回车确认、`Escape` 取消。
- 非阻塞反馈，避免频繁打断式弹窗；能内联提示就不要强制中断流程。
- 上下文连续，用户切换视图后仍能理解当前状态。
- 渐进披露，默认界面保持简洁，把次级操作放到 hover、menu、详情区或二级面板。
- 明确展示加载、空态、错误态和禁用态。
- 对危险操作给出确认或可撤销路径。
- 对长任务、流式结果或后台操作，持续展示进度和最新状态。

## Implementation Notes

- 在动手前，先说明参考了哪些外部文件或模式。
- 如果没有找到相似实现，要明确说明“未找到直接对应实现”，再基于当前项目风格设计。
- 涉及 UI 时，除了功能正确，还要检查视觉层级、入口可发现性、常用路径长度，以及是否符合编辑器类客户端的工作流节奏。

## Response Format

完成任务时，简要交代：

- 参考的 `claude-code` 文件或模式
- 最终采用的 Cursor-like 交互决策
- 如果没有照搬参考实现，说明做了哪些适配
